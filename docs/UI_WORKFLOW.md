# JobPicky 前端 UI 改造工作流

本文约束所有涉及前端视觉、布局、交互和动画的修改。目标是在不破坏现有功能和技术栈的前提下，提高设计质量，并用尽可能小、易维护的代码完成实现。

## 1. 适用范围

以下任务必须遵循本文：

- 页面、组件、布局、排版、颜色、图标和响应式样式调整；
- 导航、表单、卡片、列表、弹窗等前端交互改造；
- hover、进入/退出、时间线、滚动联动等动画；
- 用户明确提出“美化”“重设计”“提升质感”或类似要求。

纯后端、数据处理和与 UI 无关的任务不需要调用这些技能。

## 3. 规则优先级

发生冲突时，严格按以下顺序处理：

1. 用户本次明确要求和仓库 `AGENTS.md`；
2. 功能正确性、安全、可访问性和数据完整性；
3. 现有产品行为、技术栈、品牌资产和已有组件约定；
4. Taste Skills 决定视觉方向；
5. GSAP Skills 决定动画实现；
6. Ponytail 压缩代码量、依赖和改动范围。

低优先级技能不得删除或覆盖已经确定的高优先级要求。

## 4. 强制工作流

### 阶段 A：审查，暂不写代码

先调用 `redesign-existing-projects`，只读取与目标页面直接相关的模板、组件、样式、脚本和资源。

必须确认：

- 当前页面的用途、主要用户和关键任务；
- 必须保留的功能、内容、品牌元素和交互习惯；
- 现有组件、变量、工具类和依赖中可以复用的部分；
- 视觉层级、排版、间距、颜色、响应式和交互反馈中的具体问题；
- 本次修改边界，避免把局部美化扩展成全站重写。

在首次编辑前，用简短 commentary 给出：

1. 一句话 `Design Read`；
2. 主要问题；
3. 保留项；
4. 准备修改的范围。

需求只有在视觉方向确实存在分歧时才问一个澄清问题；能够从上下文可靠判断时直接继续。

### 阶段 B：选择设计技能

根据页面类型路由，不得把所有 Taste 规则无条件叠加：

- 已有产品页面、表单、列表或工作流：以 `redesign-existing-projects` 为主；
- 落地页、营销页、作品集或强视觉页面：可以同时调用 `design-taste-frontend`；
- 仪表盘、数据表格和复杂业务流程：不得机械套用营销页结构、超大留白或重型视觉效果。

Taste Skill 只提供设计判断，不得因为其默认偏好而擅自：

- 将项目迁移到 React、Next.js、Tailwind 或其他框架；
- 引入新的设计系统、图标库、字体库或动画库；
- 替换已经工作良好的组件体系；
- 把所有页面改造成同一种营销页风格。

应先设定适合当前页面的设计变化、动画强度和视觉密度。已有项目默认采取“渐进式重设计”，除非用户明确要求全面改版。

### 阶段 C：制定动画预算

任何新增或修改动画的任务，至少调用 `gsap-core` 和 `gsap-performance`；再按实际需要选择：

| 场景 | 追加 Skill |
| --- | --- |
| 多步骤编排、进入/退出序列 | `gsap-timeline` |
| 滚动触发、scrub、pin | `gsap-scrolltrigger` |
| React/Next.js | `gsap-react` |
| Vue、Nuxt、Svelte 等 | `gsap-frameworks` |
| Flip、SplitText、Draggable 等插件 | `gsap-plugins` |
| clamp、mapRange、snap 等工具 | `gsap-utils` |

动画使用原则：

- 简单 hover、focus、颜色和轻微位移优先使用 CSS transition；
- 时间线、滚动联动、复杂序列或需要精确清理的动画使用 GSAP；
- 不因“展示技术”而给每个元素添加动画；
- 优先动画 `transform` 和 `opacity`，避免频繁触发布局；
- 必须支持 `prefers-reduced-motion`；
- 组件卸载、页面切换或重新渲染时必须正确清理动画、context 和 ScrollTrigger；
- 动画不得阻塞阅读、键盘操作、表单提交和浏览器原生滚动；
- 移动端可以降低或关闭非必要动画。

GSAP 仓库是运行时库，不是 skill。实际实现前必须检查项目清单文件：

- 已安装 GSAP时复用现有版本；
- 未安装时，只有在确认 CSS 无法合理覆盖已批准效果后才添加依赖；
- 不在纯设计讨论、审查或文档阶段安装 `gsap`、`@gsap/react` 或插件；
- 安装依赖前说明原因和影响。

### 阶段 D：用 Ponytail 实现

设计方向和动画预算确定后，调用 `ponytail full`。按以下顺序选择实现：

1. 不新增代码也能解决；
2. 复用项目现有组件、样式、变量和工具；
3. 使用 CSS、HTML 和浏览器原生能力；
4. 使用已安装依赖；
5. 添加最少的新代码；
6. 只有确有必要时添加依赖或抽象。

实现约束：

- 优先最少文件和最小可工作的 diff；
- 不创建只使用一次的设计系统、wrapper、factory、helper 或配置层；
- 不复制已有样式或逻辑；
- 不为未来可能出现的需求预留脚手架；
- 删除重复或失效代码优先于叠加覆盖规则；
- 不通过 `!important`、高层级选择器或无边界 z-index 掩盖根因；
- 不得以“少写代码”为由省略可访问性、错误处理、动画清理和必要验证。

用户已经明确批准的视觉和动画目标属于需求，Ponytail 不得再次用 YAGNI 将其删除。

### 阶段 E：验证后交付

根据改动风险执行最小但充分的检查：

- 运行与目标页面相关的现有测试或构建检查；
- 在至少一个桌面宽度和一个移动宽度检查布局；
- 检查 hover、focus、键盘操作、表单和主要业务流程；
- 检查 `prefers-reduced-motion`；
- 检查横向滚动、内容截断、重叠、闪烁和布局跳动；
- 检查动画重复挂载、未清理监听器和 ScrollTrigger；
- 确认没有无必要的新依赖、重复组件和大范围无关格式化。

可以运行项目时，应使用浏览器做视觉 QA；无法运行时，必须明确说明未完成的验证，不得宣称视觉效果已经确认。

最终交付应简要说明：

- 视觉方向和实际效果；
- 修改了哪些文件；
- 使用了哪些 skills；
- 执行了哪些验证；
- 有哪些刻意未做的扩展，以及何时才值得添加。

## 5. 禁止事项

- 未审查现状就直接重写页面；
- 为追求“高级感”统一加入渐变、玻璃拟态、发光、超大圆角或卡片套卡片；
- 默认使用重型滚动劫持、pin 或视差；
- 仅因 Taste Skill 示例而更换框架或 CSS 方案；
- 同时引入多个动画库或图标体系；
- 为少量样式创建新的全局设计系统；
- 没有 reduced-motion 和清理逻辑的 GSAP 动画；
- 用大量新增代码掩盖本可复用的现有实现；
- 修改与本次 UI 目标无关的业务逻辑。

## 6. 技能更新策略

默认继续使用本文记录的提交版本。需要升级时：

1. 先阅读上游变更；
2. 确认技能名称、触发条件和工作流没有破坏性变化；
3. 更新安装内容；
4. 同步更新本文的提交哈希；
5. 用一个小型 UI 任务验证后再用于大范围改造。
