## 1. 文档用途

本文是本次 UI 改造的跨对话实施与验收基线。后续接手者应先完整阅读：

1. 仓库根目录 `AGENTS.md`；

## 2. 核心目标

在保留现有信息架构、业务行为和 FastAPI API 的前提下，把 JobPicky 从“干净的浅色 B 端工作台”升级为“冷调、通透、有真实材质层级的个人岗位雷达”。

需要同时做到：

- 参考图中的淡色环境背景、磨砂玻璃、白色高亮玻璃选中态和柔和边缘高光；
- 保留岗位列表、筛选、分页、偏好编辑、扫描状态、岗位详情和飞书集成的可读性与操作效率；
- 强化“岗位雷达”作为产品最有辨识度的视觉核心；
- 使用成熟开源实现的关键算法，不重复发明 SVG 折射与色散方案；
- 为不支持完整折射的浏览器、降低动态效果和降低透明度偏好提供可靠降级；
- 不以视觉升级为理由修改 API、数据模型、匹配算法或扫描流程。

设计参数：

| 参数 | 目标值 | 解释 |
| --- | ---: | --- |
| `DESIGN_VARIANCE` | 5 | 保留工作台结构，允许浮动导航和适度非对称层级 |
| `MOTION_INTENSITY` | 4 | 使用反馈型微动效，不做滚动劫持、视差或重型编排 |
| `VISUAL_DENSITY` | 6 | 保留岗位筛选与列表效率，不照搬参考图的展示型大留白 |

- 岗位卡信息内容、截止状态、推荐理由和详情抽屉；
- 首次设置与编辑偏好的四阶段流程；
- 雷达实时预览、聚焦度、保存、重新匹配和取消逻辑；
- 扫描、加载、空状态、错误提示、通知和重试路径；

未经单独批准，不修改 URL、API 请求、事件含义、字段名、数据结构和业务文案。

## 4. 明确不做的事项

- 不把页面改造成营销落地页；
- 不照搬参考图的冥想产品布局、文案、橙红品牌色或超低信息密度；
- 不给每一张岗位卡挂载动态 SVG 折射和独立鼠标监听；
- 不同时引入 React、GSAP、Motion、Tailwind 或新的图标库；
- 不为这次换肤创建通用组件框架、主题编辑器或完整设计系统；
- 不添加滚动劫持、磁吸光标、无限漂浮或纯装饰性动画；
- 不把上游项目整仓复制进业务源码；
- 不在旧 CSS 末尾继续叠加大块重复覆盖，需替换或整理现有规则；
- 不在本轮顺手重构后端或匹配逻辑。

## 5. 已确认的技术决策

### 5.1 默认路线：原生栈移植关键算法

继续使用现有原生 HTML/CSS/JavaScript。只移植 `liquid-glass-react` 中与效果直接相关的：

- SVG `feDisplacementMap` 折射；
- RGB 通道轻微错位产生的边缘色散；
- 边缘遮罩和中心清晰区域合成；
- `backdrop-filter` 模糊与饱和度；
- 双层边缘高光、内阴影和混合模式；
- 小范围的弹性指针反馈。

移植时保留 MIT 许可证和来源说明。建议添加：

- `src/jobpicky/web/static/js/liquid-glass.js`：尺寸、指针、特性检测和初始化；
- `src/jobpicky/web/static/css/liquid-glass.css`：玻璃材质原语和降级规则；
- `src/jobpicky/web/static/vendor/liquid-glass-react/LICENSE`：上游许可证副本。

不得照搬上游每个组件各自维护 React state 和 `mousemove` 监听的方式。当前项目应使用至多一个全局指针监听，通过 `requestAnimationFrame` 更新活动玻璃表面的 CSS 变量；非活动表面不参与逐帧更新。

### 5.2 React 决策门

本轮不默认迁移 React。只有同时满足以下条件，才允许把路线改为 Vite + React：

1. 独立材质试验确认原生移植无法达到已批准的折射质量；
2. 直接使用上游组件在目标浏览器中有肉眼可见的质量优势；
3. 用户明确批准前端架构迁移；
4. 先更新本文，写明构建目录、Python 打包衔接、迁移顺序和回滚方案；
5. React 迁移作为单独分支和单独 PR，不与当前原生换肤混在同一提交序列。

禁止采用长期存在的“原生 DOM + 多个 React 小岛共同管理同一内容节点”方案。

### 5.3 材质分级

| 层级 | 使用位置 | 实现 |
| --- | --- | --- |
| 环境层 | 页面背景 | 冷银灰底色、低饱和冰蓝与杏色光场、极轻噪点 |
| 静态玻璃 | 统计栏、岗位卡、筛选区、表单区 | 半透明背景、模糊、内外边缘、环境色阴影 |
| 高亮玻璃 | 激活 Tab、选中项、主要按钮 | 更高亮度与不透明度、白色内高光、较强层次 |
| 动态折射 | 浮动顶栏关键控件、雷达主面板、少量主操作 | SVG 位移、边缘色散、指针反馈 |
| 实色降级 | 不支持滤镜或降低透明度时 | 高对比半透明或近实色表面，不依赖背景可读性 |

动态折射不应用于岗位卡网格、长列表每一项或大量标签。

## 6. 视觉规格

### 6.1 色彩

- 保留当前钴蓝作为唯一语义强调色；
- 主背景使用冷银灰，不使用纯白；
- 环境光允许使用极低饱和冰蓝、雾紫和杏色，但它们不能成为按钮或状态色；
- 正文使用接近黑色的冷调深色，次级文字必须满足对比度要求；
- 成功、警告、错误和截止状态继续使用现有语义色。

### 6.2 形状

- 大型面板：约 `24-32px` 圆角；
- 岗位卡与普通容器：约 `18-22px`；
- 输入、Select 和普通按钮：约 `12-16px`；
- Tab、标签和高亮选中态：允许胶囊形；
- 同类组件必须统一，不得随意混用圆角。

### 6.3 排版

- 保留系统中文字体栈，不为了“高级感”增加网络字体依赖；
- 使用字号、字重、字距和留白建立层级；
- 数字统计启用等宽数字；
- 不重写业务文案，不增加装饰性英文标签和无意义小标题。

### 6.4 动效预算

- Hover、focus、active 和简单切换使用 CSS；
- 主要动画只操作 `transform`、`opacity` 和必要的滤镜参数；
- 动态折射必须能说明其作用：反馈点击、强调选中或强化雷达材质；
- `prefers-reduced-motion: reduce` 下关闭弹性、位移和非必要过渡；
- 不新增 GSAP。只有未来出现 CSS 无法覆盖的已批准编排时才重新评估。

## 8. Git 管理方案

### 8.1 开始前

```powershell
git status --short
git switch -c codex/ui-liquid-glass-redesign
```

只有在工作树状态已确认且不会覆盖用户改动时才能创建分支。若已有同名分支，先检查其来源和状态，不强制重建。

### 8.2 提交策略

每个提交必须可单独审查，尽量保持可运行：

1. `docs: add liquid glass UI redesign plan`
2. `feat(ui): add reusable liquid glass material and fallbacks`
3. `style(ui): redesign application shell and radar dashboard`
4. `style(ui): redesign preferences and secondary views`
5. `test(ui): cover glass assets and preserved web behavior`
6. `docs(ui): record visual QA and acceptance results`

根据实际改动可以合并相邻提交，但不得把后端业务修改混入 UI 提交。

每次只暂存当前阶段文件，例如：

```powershell
git add docs/LIQUID_GLASS_UI_REDESIGN_PLAN.md
git diff --cached --stat
git diff --cached -- docs/LIQUID_GLASS_UI_REDESIGN_PLAN.md
```

禁止用 `git add .` 掩盖无关改动。禁止使用 `git reset --hard`、`git checkout --` 等破坏性命令清理用户工作。

### 8.3 阶段检查点

每个阶段提交前执行：

```powershell
git status --short
git diff --stat
```

随后只检查本阶段相关文件的差异。若某阶段导致功能回归，应优先修复当前提交，不在后续提交中用覆盖规则掩盖。

### 8.4 推送与 PR

- 未经用户要求不自动推送或创建 PR；
- 需要发布时推送 `codex/ui-liquid-glass-redesign`；
- PR 描述必须引用本文，列出视觉变化、依赖变化、测试结果、浏览器降级和已知限制；
- 若最终采用 React，必须另开分支和 PR，不在本分支追加迁移。

### 8.5 回滚策略

- 材质实现和页面换肤分开提交，以便单独回滚动态折射；
- 降级样式必须与主实现同时提交，不能作为后补；
- 若动态折射导致性能或兼容问题，回滚第 2 个提交后，静态玻璃视觉仍应可继续实施；
- 不通过删除用户数据、配置或数据库完成 UI 回滚。

## 10. 最终验收标准

### 10.1 阻断级功能标准

- [ ] 首次设置可以完整创建岗位雷达并开始扫描；
- [ ] 首页统计、推荐范围切换、全部筛选、清除筛选、排序和分页正常；
- [ ] 岗位详情抽屉可打开、关闭并恢复焦点；
- [ ] 编辑偏好四个步骤、搜索、选择、删除、取消、保存和重新匹配正常；
- [ ] 扫描中、成功、失败、空数据和重试状态正常；
- [ ] 飞书集成表单与密码显示切换正常；
- [ ] 所有现有 API 路径、请求体和字段保持兼容；
- [ ] `tests/test_web.py` 与 `tests/test_ui_sandbox.py` 通过；
- [ ] 完整 `pytest` 通过，或已有无关失败被准确记录并证明非本改动引入。

### 10.2 阻断级视觉标准

- [ ] 页面背景有可感知但克制的环境色和层次，不是纯灰平面；
- [ ] 普通磨砂玻璃、白色高亮玻璃和动态折射三类层级肉眼可区分；
- [ ] 激活 Tab、选中选项和主要按钮符合“白色高亮磨砂玻璃”目标；
- [ ] 雷达是页面最强视觉焦点，但不遮挡标签和文字；
- [ ] 岗位卡没有过强反光、色散或模糊，正文可快速扫读；
- [ ] 顶栏、首页、偏好页、岗位库、详情和集成页属于同一视觉系统；
- [ ] 没有玻璃套玻璃造成的多重边框噪音；
- [ ] 没有为了高级感加入无语义发光、霓虹、超大标题或过度留白。

### 10.3 响应式标准

至少检查：

- 桌面：`1440 × 900`；
- 紧凑桌面：`1366 × 768`；
- 移动端：`390 × 844`；
- 窄移动端：`360 × 800`。

所有宽度必须满足：

- [ ] 无页面级横向滚动；
- [ ] 无内容截断、卡片重叠和按钮文字换行；
- [ ] 顶栏、筛选器、雷达、抽屉和保存操作可用；
- [ ] 移动端岗位卡保持单列，雷达不挤压主任务；
- [ ] 软键盘或窄屏下主要表单操作仍可到达。

### 10.4 可访问性标准

- [ ] 正文和控件文字达到 WCAG AA 对比度；
- [ ] 所有交互项有清晰 `:focus-visible`；
- [ ] 仅使用键盘可以完成主要流程；
