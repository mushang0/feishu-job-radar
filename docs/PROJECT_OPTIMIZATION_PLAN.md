# Feishu Job Radar 产品优化总计划

> 本文档是当前项目优化工作的唯一总纲，供后续对话和执行 Agent 读取、拆解与实施。
>
> 本计划只定义产品方向、需求边界、实施顺序和验收标准，不规定具体代码写法。每个执行任务开始前，应根据本计划进一步形成小范围技术实施方案。

## 1. 文档目标

本计划用于将 Feishu Job Radar 从“作者自己能够运行的岗位抓取脚本”，优化为：

> 面向具备基本电脑操作能力的个人求职者，通过本地运行自动发现相关岗位，并在飞书中完成筛选、收藏和投递进度管理的个人工具。

当前版本的首要目标不是达到极高的推荐准确率，而是先形成完整、稳定、容易理解的用户闭环：

```text
完成简单配置
→ 自动发现基本相关的岗位
→ 只将需要处理的岗位送入飞书
→ 用户选择收藏或不合适
→ 用户继续管理投递进度
→ 系统持续更新且不破坏用户数据
```

本文档不采用按天、按周或多个发布周期排期的方式。优化工作按照具有独立用户价值和明确验收条件的交付阶段依次完成。

## 2. 制定计划时确认的项目现状

### 2.1 仓库状态

- 项目是 Python 本地应用，目前只有一个正式提交。
- 已具备抓取、匹配、本地存储、飞书同步、状态回收、提醒和导出等基本链路。
- 当前自动化测试共 88 项，检查时全部通过。
- 测试目录目前被版本控制忽略，尚未成为公开项目质量保障的一部分。
- 用户配置文件约 300 行，混合了用户偏好、抓取参数、岗位分类、公司名单、别名和飞书凭据。
- 当前没有面向普通用户的初始化向导、配置检查和完整运行诊断。
- 当前同步流程会将本地全部岗位同步到飞书，而不是只同步推荐岗位。
- 当前飞书字段映射没有写入本地已经存在的岗位名称。
- 重点公司可以绕过岗位排除条件，导致明显不相关岗位进入推荐。
- 搜索不到可靠官方入口时，当前逻辑可能保存搜索结果页，容易造成“官方链接”名不副实。

### 2.2 本地数据状态

检查时的运行数据库包含：

- 704 条岗位；
- 704 条匹配结果；
- 150 条本地推荐记录；
- 704 条已同步标记；
- 702 条解析正常记录和 2 条部分解析记录；
- 439 条岗位没有补充到所谓官方链接；
- 本地用户状态表为空。

本地用户状态为空，而远程飞书中已经存在大量用户选择，说明飞书状态回收到本地的完整性必须作为迁移和发布前的重点检查项。这里不直接认定具体故障原因，由实施阶段验证。

### 2.3 远程飞书状态

只读检查时，当前多维表格包含：

- 708 条记录；
- 15 个字段；
- 8 个视图；
- 岗位 ID 是主字段；
- 所有记录都没有岗位名称字段；
- 152 条记录标记为推荐；
- 554 条记录标记为不推荐；
- 582 条记录为“未看”；
- 99 条记录为“不合适”；
- 26 条记录为“已收藏”；
- 4 条记录缺少公司信息；
- 443 条记录没有官方链接；
- 现有备注均为空。

远程有 708 条记录、本地有 704 条岗位，本地推荐与远程推荐数量也存在差异。迁移前必须先解释并处理这些差异，不能直接覆盖或删除远程数据。

### 2.4 对原优化建议的总体判断

原建议对以下核心问题的判断合理，应当采用：

- 飞书应是求职工作台，而不是原始数据库镜像；
- 用户只选择偏好，不维护内部知识库；
- 只同步推荐岗位；
- 岗位名称应成为主信息；
- 用户只操作一套求职状态；
- 安装、配置、飞书接入和故障检查需要简化；
- 机器人应当可选；
- 数据和同步失败必须有明确反馈；
- 测试应纳入项目并成为发布保障。

原建议中的以下内容需要简化或延后：

- 不设置“重点公司待确认”或其他人工核验状态；
- 不为用户展示高、中、低置信度；
- 不在当前版本建设复杂评分体系或反馈学习；
- 不在当前版本扩展多个数据源；
- 不在当前版本建设 OAuth、SaaS、多用户、网页配置和云端调度；
- 不为未来插件化而提前进行全面架构重写；
- 不强制提供 Docker、复杂交互卡片或完整跨平台调度能力；
- 不把所有工程工具一次性加入，优先选择直接保障当前产品的最小集合。

## 3. 产品边界

### 3.1 当前版本纳入范围

- 本地、单用户运行；
- 面向具备基本电脑操作能力的用户；
- 简化安装、初始化和配置；
- 引导用户连接自己的飞书应用和多维表格；
- 自动检查飞书权限和工作台结构；
- 保留 WonderCV 作为当前主要岗位发现来源；
- 只向飞书同步推荐岗位和用户持续跟踪的岗位；
- 在飞书中完成初筛、收藏和投递进度管理；
- 保护用户状态、下次行动和备注；
- 提供清晰的运行结果、错误提示和恢复建议；
- 建立发布所必需的测试、文档和数据迁移保障。

### 3.2 当前版本明确不做

- 网页配置界面；
- 飞书 OAuth 或飞书商店应用；
- 云端托管；
- 多用户和数据隔离；
- 多数据源聚合；
- 自动交叉核验全部岗位信息；
- 复杂推荐评分、机器学习或个性化反馈模型；
- 完整的数据源插件平台；
- 强制群机器人；
- 复杂交互式通知卡片；
- 预设库在线更新；
- 为未来设想进行与当前用户闭环无关的大规模重构。

上述内容可以记录为未来方向，但不得成为当前版本对外可用的前置条件。

## 4. 产品原则

所有后续实施和取舍必须遵守以下原则：

1. **用户任务优先**：用户只需要发现岗位和管理投递，不需要理解抓取、匹配和同步内部结构。
2. **少选择、少指令**：能采用安全默认值的地方，不要求用户做决定；不为每个小功能增加命令。
3. **飞书只展示有用内容**：原始岗位库保留在本地，飞书只展示推荐或已被用户跟踪的岗位。
4. **单一用户状态**：用户只操作一套求职状态，内部状态默认隐藏。
5. **不虚构信息可靠性**：不确定的信息保持未知，不把搜索结果页描述为官方入口。
6. **先可用，再优化准确率**：推荐质量可以持续调整，但不能阻塞基本产品闭环。
7. **用户数据不可被自动化破坏**：状态、下次行动、备注及已跟踪岗位必须得到保护。
8. **失败必须可见**：不能出现程序显示成功、但岗位没有同步到飞书且用户不知情的情况。
9. **渐进改造**：每个阶段都应产生可以独立验收的结果，不进行一次性推倒重来。
10. **避免过度设计**：任何新增功能都应能明确回答它解决了当前哪个用户问题。

## 5. 目标用户流程

### 5.1 首次使用

用户应完成以下最短流程：

1. 启动安装或初始化流程。
2. 选择毕业届别、岗位方向和目标城市。
3. 按需选择公司类别、重点公司和少量补充关键词。
4. 提供飞书应用凭据和多维表格链接。
5. 系统检查配置、访问权限和工作台结构。
6. 系统创建或调整所需字段和视图。
7. 系统完成首次岗位扫描和同步。
8. 用户打开飞书后能够立即看到待处理岗位。

首次使用不应要求用户：

- 理解爬虫页数、间隔、重试等参数；
- 阅读或修改数百行配置；
- 维护公司别名和岗位关键词库；
- 手动创建十多个飞书字段；
- 手动设计筛选条件和视图；
- 理解推荐状态、核验状态、解析状态或同步状态；
- 必须配置群机器人。

### 5.2 日常使用

日常流程应固定为：

1. 系统发现新岗位；
2. 系统在本地保存岗位和匹配信息；
3. 系统只将符合基本条件的新岗位同步到飞书；
4. 新岗位统一进入“待处理”；
5. 用户将其改为“收藏”或“不合适”；
6. 收藏后，用户可继续改为“已投递”“笔试中”“面试中”“Offer”或“已结束”；
7. 系统后续更新岗位信息，但不覆盖用户维护的信息。

### 5.3 求职状态

用户只操作以下状态：

```text
待处理
收藏
不合适
已投递
笔试中
面试中
Offer
已结束
```

明确取消或隐藏：

- 重点公司待确认；
- 用户可见的推荐状态；
- 用户可见的核验状态；
- 用户可见的同步状态；
- 高、中、低置信度；
- 需要用户理解的多套并行状态。

重点公司岗位如果满足进入工作台的基本条件，与其他岗位一样进入“待处理”。用户自行选择收藏或不合适。

## 6. 飞书工作台设计

### 6.1 工作台职责

飞书只负责：

- 展示需要用户处理的推荐岗位；
- 保存用户的筛选结果和投递进度。

全部抓取数据、匹配记录、去重信息、解析状态和同步细节保留在本地系统中。

### 6.2 岗位进入和保留规则

- 新岗位只有满足当前基本推荐条件时才进入飞书；
- 所有新推荐岗位统一为“待处理”；
- 不推荐岗位仅保存在本地；
- 用户已经收藏或进入投递流程的岗位必须持续保留；
- 用户标记为“不合适”的岗位不出现在主要视图中；
- 重新匹配不得让已处理岗位重新变成“待处理”；
- 不再符合推荐条件的待处理岗位可以退出主要视图，但不能在没有保护措施的情况下直接删除；
- 同一次扫描或后续扫描不得重复创建同一岗位；
- 单次同步失败不得损坏或清空已有记录。

### 6.3 用户字段

工作台保留以下用户可理解字段：

| 字段 | 用途 |
|---|---|
| 公司 | 主字段，快速识别招聘企业 |
| 岗位 | 普通信息字段；不作为主标题以降低提取异常的影响 |
| 城市 | 判断地点是否合适 |
| 届别 | 判断招聘对象 |
| 批次 | 实习、提前批、秋招等 |
| 投递入口 | 访问岗位或投递页面的统一入口 |
| 截止时间 | 已知时展示，未知时留空 |
| 求职状态 | 用户唯一需要维护的状态 |
| 备注 | 用户自由记录 |

飞书只创建和同步上述字段；所有字段都面向用户显示。

### 6.4 默认隐藏的内部字段

以下信息仅保留在本地 SQLite，不创建或同步到飞书：

- 内部岗位 ID；
- 推荐是否仍有效；
- 匹配分数；
- 核验状态；
- 解析状态；
- 数据来源标识；
- 推荐理由和来源详情链接；
- 首次发现时间；
- 最后更新时间；
- 下次行动、同步记录 ID；
- 同步状态；
- 匹配规则版本。

### 6.5 投递入口

只向用户突出一个链接字段：“投递入口”。

选择原则：

1. 有明确、可信的企业投递页面时使用该页面；
2. 没有可靠官方页面时使用 WonderCV 来源详情页；
3. 搜索结果页不得标记为官方投递入口；
4. 无法确认链接性质时显示为“查看来源”，不得暗示已经核验；
5. 项目应统一提示用户投递前以企业官方招聘信息为准。

来源链接和推荐理由不写入飞书；用户只看到统一的投递入口。

### 6.6 默认视图

#### 待处理

筛选条件：求职状态为“待处理”。

主要显示：公司、岗位、城市、届别、批次、投递入口、截止时间、求职状态、备注。

排序时让新发现岗位优先，并让已知即将截止的岗位更醒目。该视图不限定“仅今日”，避免昨天未处理的岗位第二天消失。

#### 收藏

筛选条件：求职状态为“收藏”。

使用画册视图，主要显示：公司、岗位、城市、投递入口、截止时间、求职状态、备注。

公司是画册卡片主标题，不显示岗位 ID 或其他内部信息。

#### 投递进度

按求职状态分组，显示除“不合适”外的全部记录。

使用看板视图，主要显示：公司、岗位、投递入口、截止时间、求职状态、备注。

“不合适”不建立默认用户视图。可以保留一个默认隐藏的全部记录视图，用于排查和恢复数据。

### 6.7 同步保护

- 系统可以更新岗位名称、公司、城市、届别、批次、截止时间和链接；
- 系统不得覆盖用户设置的求职状态和备注；
- 系统必须按本地保存的飞书 record ID 对应记录，不能依赖表格行顺序或飞书隐藏字段；
- 飞书中的用户状态必须能够安全回收到本地；
- 用户已经跟踪的岗位不因重新匹配而消失；
- 同步失败时保留待同步状态，后续可以安全重试；
- 运行结果必须区分新建、更新、跳过和失败数量。

## 7. 当前飞书表格迁移要求

迁移属于高风险操作，必须在工作台结构和同步规则验证通过后执行。

### 7.1 迁移原则

1. 先导出并保存完整远程备份；
2. 先核对远程 708 条记录与本地 704 条岗位的差异；
3. 先验证远程用户状态能完整回收到本地或迁移数据中；
4. 保留所有用户已经处理过的状态；
5. 将“已收藏”统一迁移为“收藏”；
6. 将仍然有效的推荐岗位加入新工作台；
7. 将非推荐且未处理的历史记录退出默认视图；
8. 隔离空白、重复和无法关联的记录；
9. 将主字段从岗位 ID 改为岗位名称；
10. 迁移过程中不直接删除唯一副本；
11. 迁移完成并核对无误后，再决定是否清理旧结构。

### 7.2 迁移数据保护基线

检查时远程已有 125 条明确用户判断结果，包括 99 条“不合适”和 26 条“已收藏”。迁移后的对应数量和岗位关联必须完整保留。

### 7.3 迁移验收

- 迁移前存在可恢复的完整备份；
- 所有远程记录都有明确的迁移结果：保留、隐藏、隔离或确认清理；
- 已处理岗位数量不减少；
- 用户状态与岗位对应关系不发生错位；
- 岗位名称成为主字段；
- 默认用户视图中不再展示全量非推荐岗位；
- 不存在明显重复岗位；
- 无法关联的记录单独列出，不静默丢弃；
- 新旧数量差异有书面核对结果。

## 8. 配置与首次初始化

### 8.1 配置分层

配置应分为三类概念，但普通用户只接触第一类和必要连接信息。

#### 用户偏好

- 毕业届别；
- 目标岗位方向；
- 目标城市；
- 公司类别；
- 可选重点公司；
- 少量额外包含关键词；
- 少量额外排除关键词；
- 运行时间或是否自动运行。

#### 项目内置知识

- 岗位分类及关键词；
- 公司类别及预设公司；
- 公司别名；
- 默认排除岗位；
- 常用城市和招聘批次；
- 安全的抓取和同步默认值。

这些内容由项目维护，不要求每个用户复制和维护。

#### 高级设置

抓取页数、访问间隔、超时、重试、日志等设置采用安全默认值。只有确有需要的高级用户才修改，不进入普通首次流程。

### 8.2 首次初始化应完成的事情

- 收集最少量用户偏好；
- 检查输入是否有效；
- 引导填写或保存飞书连接信息；
- 从用户提供的多维表格链接中提取可识别信息；
- 检查应用身份和多维表格访问权限；
- 检查或建立所需字段；
- 检查或建立三个默认视图；
- 检查本地数据目录是否可用；
- 执行一次不破坏数据的连接验证；
- 在用户确认后执行首次扫描；
- 给出清晰的完成结果和下一步入口。

### 8.3 决策最小化

首次初始化原则上只要求用户决定与个人求职需求直接相关的内容。以下事项采用默认方案，不作为普通用户问题：

- 数据库位置；
- 抓取页数；
- 请求间隔；
- 重试次数；
- 字段类型；
- 视图筛选方式；
- 是否建立内部同步字段；
- 推荐分数阈值；
- 是否导入完整历史数据库。

历史岗位默认不应悄悄全部导入。首次版本可以采用“从当前有效数据开始”或一个项目明确规定的安全默认范围，不要求用户在多个历史范围中选择。

### 8.4 凭据安全

- 示例配置不得包含真实凭据；
- 公开仓库不得跟踪个人飞书凭据；
- 日志和错误提示不得输出完整密钥、访问令牌和机器人地址；
- 文档必须解释凭据用途和最小权限；
- 凭据缺失或权限不足时应明确指出问题，而不是继续运行并静默跳过。

### 8.5 初始化验收

- 普通用户不需要编辑约 300 行配置；
- 用户只填写求职偏好和必要的飞书连接信息；
- 用户无需手动创建飞书字段和视图；
- 无效配置在开始抓取前被发现；
- 权限不足能得到可理解的修复说明；
- 初始化重复执行不会创建重复字段、视图或测试记录；
- 初始化完成后能够直接打开可用的“待处理”视图；
- 未配置机器人不影响主要功能使用。

## 9. 推荐与岗位筛选的当前目标

### 9.1 当前定位

当前推荐的目标是提供“基本相关、可以快速人工筛选”的候选岗位，不追求一次性替用户做出最终判断。

推荐系统负责减少明显无关岗位，但最终由用户通过“收藏”或“不合适”完成判断。

### 9.2 当前必须修正的规则

- 明确排除的岗位不能因为公司是重点公司而直接进入推荐；
- 重点公司只是一个正向参考，不代表该公司的全部公告都相关；
- 岗位标题和明确岗位标签应比公告全文中的泛化词更可信；
- 城市、届别和招聘批次的明确不匹配应得到一致处理；
- 信息缺失与明确不匹配应当区分，避免因为数据源缺失而过度删除岗位；
- 推荐理由必须使用普通用户能理解的短句；
- 不再生成“重要公司公告，岗位方向需人工确认”这类用户状态。

### 9.3 推荐理由

推荐理由只用于解释，不要求用户理解分数。优先表达：

- 命中的岗位方向；
- 命中的明确关键词；
- 目标公司；
- 目标城市；
- 届别或批次符合。

推荐理由应短、稳定、可读，不直接展示内部调试信息。

### 9.4 当前不做的推荐能力

- 不建设复杂多阶段评分模型作为上线前提；
- 不要求用户填写“不合适原因”；
- 不根据用户状态自动学习个人权重；
- 不进行大规模官方站点聚合；
- 不承诺推荐岗位信息已经核验；
- 不用推荐准确率目标阻塞工作台、配置和可靠性改造。

### 9.5 推荐验收

- 明确销售、市场、运营、人力、财务等排除岗位不会因重点公司规则绕过排除；
- 新推荐全部进入同一个“待处理”流程；
- 用户看不到“待核验”或“重点公司待确认”；
- 每个推荐岗位有简短可读的推荐理由；
- 重新匹配不会破坏用户已做出的选择；
- 推荐不足或误报时，用户仍可以用一次状态修改完成处理。

## 10. 信息来源与可信度

### 10.1 当前原则

WonderCV 继续作为岗位发现来源，但项目不得暗示其信息完全准确。

- 工作台应能追溯岗位来源；
- 无法确认的发布时间、截止时间或公司信息保持为空或未知；
- 不用抓取日期冒充岗位发布时间；
- 不把搜索结果列表页保存成官方招聘页面；
- 没有可靠官方入口时，允许用户查看来源详情；
- 产品文档和工作台统一提醒用户投递前核对企业官方信息。

### 10.2 后续方向

以下内容列为未来优化，不属于当前完成条件：

- 为重点公司补充官方招聘入口；
- 支持用户手动补充或纠正投递链接；
- 对关键字段进行额外检索；
- 引入第二数据源交叉验证；
- 展示信息更新时间和可信来源；
- 建立错误信息反馈机制。

### 10.3 验收

- 工作台不出现将搜索结果页包装成官方链接的情况；
- 来源页和确认过的投递入口在表达上能够区分；
- 缺失日期不使用不相关日期填充；
- 用户能够知道信息需要在投递前核验；
- 信息补充失败不会阻塞岗位进入待处理流程。

## 11. 日常运行、诊断与通知

### 11.1 用户需要的主要操作

产品应围绕少量主要操作组织，不为每个内部步骤暴露独立命令。用户真正需要的是：

- 首次初始化；
- 运行一次岗位扫描；
- 检查当前配置和连接是否正常；
- 在偏好变化后重新匹配；
- 查看最近一次运行结果；
- 可选安装或说明每日自动运行方式。

具体命令名称由实施方案决定，但用户文档必须给出一条清晰主路径，不能让用户在大量命令中自行组合工作流。

### 11.2 运行前检查

至少检查：

- 配置是否完整有效；
- 本地数据目录是否可读写；
- WonderCV 是否可以访问；
- 页面是否仍能解析出有效岗位；
- 飞书身份认证是否成功；
- 多维表格是否可读写；
- 工作台字段和视图是否符合要求；
- 用户状态是否能够读取；
- 自动运行环境是否可以找到项目和配置。

### 11.3 运行结果

每次运行应给用户一份简洁结果，至少包含：

- 是否成功完成；
- 发现多少岗位；
- 新增多少岗位；
- 推荐多少岗位；
- 飞书新建、更新、跳过和失败多少条；
- 用户状态是否成功回收；
- 是否存在数据源或同步异常；
- 出现问题时用户下一步应做什么。

### 11.4 结果状态

运行不能仅以抓取是否失败判断成功。结果至少应表达：

- 完整成功；
- 部分成功但存在可恢复问题；
- 主要功能失败。

如果岗位已抓取但飞书同步失败，不能向用户报告为完整成功。

### 11.5 通知

- 群机器人保持可选；
- 未配置机器人不产生警告噪音或影响成功状态；
- 当前只需要简洁汇总，不要求交互卡片；
- 通知不堆叠大量长链接；
- 飞书工作台是主要入口，通知只是提醒；
- 截止时间提醒和下次行动提醒可以后续增强，不作为当前完成条件。

### 11.6 运行验收

- 新用户能明确知道运行是否成功；
- 飞书失败不会被隐藏在“成功”结果中；
- 重复运行不会重复创建岗位；
- 中断或部分失败后可以安全重试；
- 页面结构明显变化、抓取结果异常为空时能够被发现；
- 日志不泄露凭据；
- 没有机器人时完整核心流程仍然成立。

## 12. 数据安全、备份与升级

### 12.1 数据所有权

本地数据库是原始岗位和系统记录的主要存储，飞书是用户工作台。两者都有需要保护的信息，不能简单地认为一方可以随时覆盖另一方。

### 12.2 用户数据保护

以下内容属于用户数据：

- 求职状态；
- 下次行动；
- 备注；
- 用户手动修正的投递入口；
- 已收藏或已进入投递流程的岗位集合。

任何迁移、重新匹配或同步都不得静默覆盖这些内容。

### 12.3 备份与恢复

- 远程工作台迁移前必须备份；
- 本地数据结构升级前必须具备恢复方案；
- 备份结果应有日期和来源说明；
- 恢复流程必须至少经过一次验证；
- 不把未经说明的历史种子库静默复制为用户当前数据；
- 发布时应说明历史数据的生成时间和适用范围。

### 12.4 升级兼容

- 旧配置应得到明确迁移提示；
- 旧飞书字段应能够识别并安全迁移；
- 重复初始化不得破坏已有工作台；
- 升级失败时保留旧数据和恢复路径；
- 不能要求用户通过手工改数据库完成常规升级。

### 12.5 验收

- 用户数据保护范围有自动化验证；
- 飞书迁移和本地升级都有备份；
- 旧数据不会因字段重命名而错位；
- 失败后能够回到迁移前状态；
- 种子数据来源和时效性透明；
- 重复执行初始化和升级操作是安全的。

## 14. 实施顺序

以下顺序是依赖关系和交付顺序，不是时间周期。后续 Agent 应一次只领取一个阶段，先形成该阶段的技术方案，再实施和验收。

### 阶段 1：冻结产品规则并建立安全基线

目标：在修改用户数据和远程结构前，先明确不可破坏的行为。

交付要求：

- 固定本计划中的产品范围、用户状态和同步原则；
- 将现有测试纳入版本管理；
- 建立当前本地数据库和远程飞书的基线报告；
- 验证远程状态回收到本地的行为；
- 建立飞书和本地数据备份方案；
- 为用户状态不被覆盖、岗位不重复和失败可重试建立验证。

阶段验收：

- 当前 125 条远程用户判断能够被准确识别；
- 远程 708 条与本地 704 条的差异有明确说明；
- 后续修改具备可恢复备份；
- 核心数据保护规则有自动化验证；
- 未对远程飞书进行不可逆修改。

### 阶段 2：建立推荐岗位同步边界

目标：停止把全量原始岗位同步到用户工作台。

交付要求：

- 新增同步只处理推荐岗位；
- 已收藏或进入投递流程的岗位持续保留；
- 不再推荐但尚未处理的岗位安全退出主要视图；
- 重点公司不能绕过明确岗位排除；
- 新岗位统一使用“待处理”；
- 取消用户可见的人工核验分支；
- 同步结果能够清楚报告。

阶段验收：

- 非推荐岗位不会新增到飞书；
- 推荐岗位能够稳定新增和更新；
- 用户字段不被覆盖；
- 重复运行不创建重复记录；
- 重新匹配不破坏已处理岗位；
- 飞书失败不会显示为完整成功。

### 阶段 3：重建飞书求职工作台

目标：让飞书围绕用户任务工作，而不是展示内部数据库字段。

交付要求：

- 岗位名称成为主字段；
- 建立“待处理、收藏、投递进度”三个默认用户视图；
- 建立单一求职状态；
- 增加推荐理由、投递入口、截止时间、下次行动等必要字段；
- 默认隐藏内部字段；
- 合并用户可见的链接入口；
- 按迁移要求处理现有 708 条记录。

阶段验收：

- 用户打开飞书即可理解每个视图要完成的任务；
- 岗位卡片或表格首要信息是岗位名称和公司；
- 已收藏和不合适状态完整保留；
- 默认视图不再包含大量非推荐岗位；
- 没有“重点公司待确认”视图或状态；
- 迁移结果可核对、可恢复。

### 阶段 4：简化配置和首次初始化

目标：让新用户不再通过维护大型配置文件和手工建表开始使用。

交付要求：

- 用户配置只保留个人偏好和必要连接信息；
- 岗位分类、公司预设和别名由项目维护；
- 提供有引导的首次初始化；
- 自动检查配置、权限、字段和视图；
- 初始化可以安全重复执行；
- 机器人保持可选；
- 凭据得到安全处理。

阶段验收：

- 普通用户无需修改大型配置；
- 无需手动创建飞书字段和视图；
- 配置错误在开始抓取前被发现；
- 初始化结束后能够直接使用待处理视图；
- 重复初始化不产生重复结构或数据；
- 示例和日志不泄露凭据。

### 阶段 5：完善日常运行和故障诊断

目标：让用户知道系统是否正常、哪里失败以及如何恢复。

交付要求：

- 提供少量清晰的主要操作入口；
- 增加完整运行前检查；
- 输出抓取、推荐、状态回收和同步结果；
- 区分完整成功、部分成功和失败；
- 页面结构异常或连续无数据时给出警示；
- 自动运行方式有清晰文档；
- 通知失败不影响工作台主流程。

阶段验收：

- 普通用户能够理解运行结果；
- 同步失败不会被静默忽略；
- 中断后可安全重试；
- 自动运行能够找到正确配置和数据；
- 没有机器人时仍能完整运行；
- 日志足以排查问题且不包含秘密信息。

### 阶段 6：完成对外发布质量

目标：使项目可以交给另一位目标用户按照文档独立安装和使用。

交付要求：

- 完成端到端回归验证；
- 完善真实页面解析回归材料；
- 修正文档、项目元数据和能力描述；
- 提供许可证、限制说明和数据来源声明；
- 使用全新环境和全新配置完成一次用户验收；
- 记录升级和恢复方法。

阶段验收：

- 一位没有参与开发的目标用户能够依照文档完成使用；
- 用户不需要开发者口头补充关键步骤；
- 首次配置、扫描、同步、收藏和投递状态更新完整通过；
- 重新运行和重新匹配不会破坏用户数据；
- 当前已知限制被准确表达；
- 项目达到本计划第 15 节的整体完成标准。

### 后续优化：推荐质量和信息可信度

只有在前六个阶段完成后，再根据真实用户反馈决定是否实施：

- 增加官方招聘入口；
- 支持手动补充和纠正信息；
- 增加第二信息来源；
- 优化字段权重和推荐排序；
- 记录不合适原因；
- 根据反馈调整个人偏好；
- 增加截止时间和行动提醒。

这些内容不得反向扩大当前版本范围。

## 15. 整体完成标准

当且仅当以下条件全部满足时，当前优化计划才算完成：

### 用户体验

- 新用户只需提供个人求职偏好和必要飞书连接信息；
- 用户不需要理解内部字段、匹配规则或同步结构；
- 用户不需要手动创建飞书字段和视图；
- 用户日常只需处理待处理、收藏和投递进度三个工作区；
- 初筛一个岗位只需选择“收藏”或“不合适”；
- 没有“重点公司待确认”等额外决策负担。

### 飞书工作台

- 岗位名称是主字段；
- 默认用户视图为待处理、收藏和投递进度；
- 默认只展示推荐和持续跟踪的岗位；
- 推荐理由清晰可读；
- 投递入口不会虚构官方可靠性；
- 用户状态、下次行动和备注不会被覆盖。

### 运行可靠性

- 重复扫描不会重复创建岗位；
- 状态可以从飞书安全回收到本地；
- 重新匹配不会破坏用户选择；
- 同步失败有明确结果和恢复建议；
- 中断后能够安全重试；
- 数据迁移和升级有备份与恢复能力。

### 推荐基本质量

- 明确排除岗位不会因重点公司规则被推荐；
- 所有推荐进入同一个待处理流程；
- 每个推荐有简洁理由；
- 信息缺失不会被伪装成已经核验；
- 推荐准确性仍可后续调整，但不妨碍用户完成筛选。

### 对外可用性

- 测试和核心质量保障已纳入仓库；
- 新环境能按照公开文档完成端到端运行；
- 示例不含真实凭据；
- 文档只描述实际存在的能力；
- 数据来源、准确性限制和隐私边界有明确说明；
- 一名目标用户可以不依赖开发者完成首次使用。

## 16. 后续 Agent 使用规则

其他对话或 Agent 实施本计划时，应遵循：

1. 开始前完整阅读本文档；
2. 一次只领取一个实施阶段或其中一个可独立验收的子任务；
3. 先检查仓库和远程飞书的最新状态，不能只依赖本文档中的快照数量；
4. 在改动前说明本次任务对应的目标、范围和验收条件；
5. 根据当前代码形成具体技术实施方案，但不得改变本计划的产品边界；
6. 涉及远程飞书、迁移、删除或覆盖数据时，必须先备份并单独确认；
7. 不得以架构优化为理由引入当前范围外的 SaaS、多用户、复杂推荐或插件平台；
8. 每项改动必须验证用户数据不会被覆盖；
9. 完成后按本文档中的阶段验收标准报告结果；
10. 如发现计划与真实状态冲突，应先记录事实和影响，再请求调整计划，不得静默改变产品方向。

```text
本次范围：
明确不做：

遗留问题：
```

## 17. 已确认的关键决策

- 采用“用户闭环优先”路线；
- 当前版本是本地、单用户 Community 形态；
- 面向具备基本电脑操作能力的个人用户；
- 不按时间周期拆分优化工作；
- 先达到可面向用户，再优化推荐准确性；
- WonderCV 信息可能不准确，当前通过来源透明和投递前核验提示处理；
- 以后可以增加其他来源或信息检索，但不属于当前前置条件；
- 不设置“重点公司待确认”；
- 所有进入工作台的新岗位统一为“待处理”；
- 用户将待处理岗位改为“收藏”或“不合适”；
- 不让用户参与过多内部决策；
- 不增加过多命令；
- 产品优化总计划固定保存在 `docs/PROJECT_OPTIMIZATION_PLAN.md`。
